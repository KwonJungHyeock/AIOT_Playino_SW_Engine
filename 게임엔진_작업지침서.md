## 1. 왜 게임엔진인가 / 무엇을 만드나
현재 미니게임은 Canvas 2D로 직접 그려져 있다(예: DHT11 "쾌적 지키기"). 더 풍부한 상호작용·연출·물리·타일맵이 필요한 방들을 위해 **Phaser 3**(MIT, 무료)로 미니게임을 제작한다.

- **엔진 권장**: **Phaser 3** (MIT 라이선스, 완전 무료, 상업적 사용 가능). 대안: PixiJS(MIT, 렌더러), Kaplay/Kaboom(MIT). **유료 엔진(Unity/GameMaker 등) 사용 금지** — 본 제품은 서버리스(Vercel) + 접속코드 판매 구조라 런타임 라이선스 비용이 없어야 함.
- **번들**: 각 미니게임은 **독립 ES 모듈**로 빌드돼 셸에서 동적 import 된다. 엔진은 게임 모듈 내부에 캡슐화(셸 전역에 Phaser를 두지 않음) → 코드 스플리팅으로 초기 로딩 보호.
- **산출물**: 방 1개 = 미니게임 모듈 1개. §6 표의 메커니즘을 구현.

## 2. 기술 스택 & 리포 구조 (현행)
- **빌드**: Vite 5, 바닐라 JS(프레임워크 없음). 진입 `index.html` → `src/main.js`.
- **씬 전환**: `#app` 의 `innerHTML` 을 교체하는 수동 씬 매니저(`src/main.js`). 라우터 없음.
- **하드웨어**: WebSerial(크롬 계열) + STK500v1 웹 플래싱. 펌웨어는 ATmega328P(아두이노 UNO) 베어메탈.
- **에디터**: CodeMirror 6(코드), Blockly 11(블록) — 학습 코딩용. 공용 인터프리터(`src/editor/interpreter.js`)가 판정/실행.
- **폰트**: Orbitron / Space Grotesk (Google Fonts).
- **배포**: Vercel 정적 호스팅. 인증은 접속코드(서버리스). **백엔드 없음** → 모든 상태는 `localStorage`.

핵심 디렉터리:
```
src/

    progress.js           # ★ 진척 저장(localStorage). 클리어 여부/집계
```

## 3. 학습 흐름 & 내비게이션 (셸이 제공, 너는 L2만)
3단 구조:
- **L0 · 연구소 복도(HUB)**: 4개 **챕터 입구(게이트)**. 어둡고, 클리어할수록 밝아짐(탈출 진척 시각화).
- **L1 · 챕터 방**: 입구로 들어가면 **하위 미션 방 문**들이 보임(예: CH2 → 15개).
- **L2 · 미션 방**: 문으로 입장 → **학습(체험→블록/코드→업로드) + 미니게임**. ← **너의 작업 영역**.

미션 방 1개의 학습 사이클(기존 규약, 유지):
1. **체험(버튼)** → 2. **블록/코드 도전(판정)** → 3. **업로드로 동작 확인** → 4. **미니게임(탈출 퍼즐)** → 5. **클리어 → 탈출 보상**.

## 4. ★ 데이터 계약 (셸 ↔ 미니게임) — 가장 중요
미니게임 모듈은 아래 시그니처를 **반드시** 구현한다. 셸이 방 진입 시 동적 import 하여 호출한다.

```js
// src/games/<roomId>.js  (예: src/games/buzzer.js)
//
// root : 미니게임을 렌더할 DOM 컨테이너(셸이 비워서 전달). 전체 영역 사용 가능.
// ctx  : 아래 GameContext.
// 반환 : { destroy() }  — 셸이 방을 나갈 때 호출(타이머/이벤트/엔진 정리 필수).
export function mountGame(root, ctx) {
  // ... Phaser 게임 생성 ...
  return { destroy() { /* game.destroy(true); 리스너 해제 */ } };
}
```

### GameContext (셸 → 게임)
```ts
ctx = {
  mission: {
    id: string,            // 방 고유 id (예: 'buzzer') — progress 키
    chapter: 'ch1'|'ch2'|'ch3'|'ch4',
    name: string,          // '부저'
    icon: string,          // '🔊'
    concept: string,       // 학습 개념 (예: 'tone/주파수')
    escape: string,        // 탈출 상황 프레이밍 (예: '경보 해제')
    reward: string,        // 클리어 보상 (예: '정적 회복')
    pins: number[],        // 사용 핀 (예: [8]) — 하드웨어 제어 대상
    learning: object,      // 미션/힌트/난이도 등 세부(있으면). 명세서_4챕터_상세.md 참조
  },
  board,                   // ★ §5 공유 보드 컨트롤러(연결됐을 때만 실물 반영)
  progress,                // 읽기 전용 진척 조회: progress.isCleared(id) 등
  say(text, x?, y?),       // EDDIE 말풍선(가이드 멘트). 좌표 생략 시 중앙 상단.
  // --- 콜백 (게임 → 셸) ---
  onComplete(detail?),     // ★ 탈출/클리어 성공 시 1회 호출 → 셸이 progress.mark(id) + 보상/폭죽
  onProgress(ratio),       // 0~1 진행률(선택) → 셸 HUD/헤더 갱신용
  onFail(reason?),         // 실패(선택). 셸은 재시도 유도(치명적 아님)
  onExit(),                // 방을 떠나 챕터로 복귀
}
```

규칙:
- **성공 판정은 게임이 한다.** 셸은 `onComplete` 한 번을 신뢰하고 진척을 기록한다. 중복 호출 금지(가드).
- **하드웨어는 best-effort**: `board.connected === false`면 화면으로만 연출하고 그대로 클리어 가능해야 한다(오프라인 학습 보장). 보드 없다고 진행을 막지 마라(과거 버그 원인).
- 게임은 `root` 밖 DOM을 만들지 않는다(모달은 셸 공용 클래스 재사용 권장). `destroy()`에서 **모든** rAF/interval/이벤트/Phaser 인스턴스를 해제.
- 좌표·연출은 자유지만 **K-12 안전 공포 톤**(§7) 준수.

### 진척 저장 (`src/app/progress.js`)
- 키: `localStorage['eduino.progress.v1']` = 클리어한 방 id의 배열(Set 직렬화).
- API: `progress.isCleared(id)`, `progress.mark(id)`, `progress.unmark(id)`, `progress.reset()`.
- 챕터/전체 집계는 `content/curriculum.js` 헬퍼가 이 진척을 읽어 계산(§8). **게임은 progress.mark를 직접 부르지 말고 `onComplete`만 호출**(셸이 마킹) — 단일 책임.

## 5. ★ 하드웨어 계약 — `src/app/board.js` (싱글턴)
미니게임은 이 객체로만 하드웨어를 만진다. (직접 SerialPort 열지 마라.)

| 멤버 | 설명 |
|---|---|
| `board.connected` | `boolean` — 포트 열림 + 핸드셰이크 OK일 때만 true |
| `board.digital(pin, on)` | `await` — digitalWrite(HIGH/LOW). 1.2s 타임아웃 보호 |
| `board.pwm(pin, v)` | `await` — analogWrite(0~255) |
| `board.readDht({timeout})` | `Promise<{temp,hum}|null>` — DHT11 1회 읽기(펌웨어 v2 필요) |
| `board.blink(pin, times, period)` | 깜빡임(항상 OFF로 끝) |
| `board.flash({onProgress,onLog})` | 웹 펌웨어 재플래싱(고급. 보통 불필요) |
| `board.onLine(fn)` / `board.onState(fn)` | 시리얼 라인/연결상태 구독(해제 함수 반환) |
| `board.log(kind, text)` | 시리얼 모니터에 로그 |

- 와이어 포맷/응답 코드는 `src/serial/protocol.js` 참조(`DHT:온도,습도`, `ERR:dht`, PING/버전 핸드셰이크 등).
- **새 센서(부저/매트릭스/스텝모터 등)를 실제 제어하려면 펌웨어 명령 추가가 필요**할 수 있다. 펌웨어 변경은 별도 합의 사항 — 우선 **시뮬레이션(화면) 우선**으로 구현하고, 실물 제어는 `board.connected`일 때만 best-effort로 얹어라. 펌웨어 확장이 필요하면 §10 백로그에 적고 진행 전 확인.

합계: **1 + 15 + 5 + 1 = 22방** (헤더 "전체 진척 N/22"와 일치).

## 7. 톤 · 연출 · 에셋 가이드 (K-12 안전 공포)
- **톤**: 으스스한 미스터리(폐연구소, 깜빡이는 비상등, 정적, 낮은 험, 먼지·그림자). **고어/폭력/유혈/인물 위협/실제 공포 연출 금지.** 위협은 추상 장치(봉인·타이머·어둠·침수)로만.
- **EDDIE**: 용감·호기심. 어둠 속에서 **EDDIE 눈 글로우가 유일한 빛**(시그니처). 클리어 시 안도·성취.
- **조명**: 미해결=어둡고 EDDIE 눈빛만. **클리어 시 그 구역만 환하게**(전력 복구 연출). HUB는 풀린 방 수만큼 점점 밝아짐.
- **색**: 다크 네이비~블랙 + **비상등 앰버(#ffb020)/레드(#ff5a3c) 포인트 글로우**, 깜빡임, 긴 그림자, 먼지 입자, 비네팅. 성공 강조색은 그린(#3ddc91), 정보 블루(#6fb7ff), 경고 옐로(#ffd11a).
- **사운드(옵션)**: 낮은 험, 멀리 삐걱, 클리어 차임. 구현 시 mute 토글 제공.
- **폰트/스타일**: 셸과 통일(Orbitron/Space Grotesk, 글래스/글로우/비네트). `src/styles/main.css` 클래스 재사용 권장.

## 8. 셸 통합 방법 (게임을 방에 꽂기)
1. 게임 모듈을 `src/games/<roomId>.js` 로 두고 `mountGame(root, ctx)` 내보낸다.
2. 셸의 미션 방 씬이 진입 시 `const { mountGame } = await import('../games/<roomId>.js')` 로 로드하고 컨테이너+`ctx`를 넘긴다(셸 측 어댑터는 본 워크스페이스가 제공/연결).
3. 클리어 시 `ctx.onComplete()` → 셸이 `progress.mark(mission.id)` + 탈출 보상/폭죽 + 챕터·HUB 밝아짐.
4. `destroy()`에서 정리. 메모리 누수/유령 타이머 금지.
- **빌드 분리**: 미니게임은 동적 import로 코드 스플리팅(초기 번들 보호 — 현재 메인 번들이 이미 큼).
- **데이터 출처**: 방 메타데이터는 `src/content/curriculum.js`(셸 신설)에서 단일 공급. 게임은 자기 `mission` 객체만 받는다(전역 의존 금지).
